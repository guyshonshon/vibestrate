# Design: Merge Window + Supervisor Merge Advisor (T13)

Status: **proposed - adversarially reviewed** · Triage:
[`backlog-triage-2026-06.md`](./backlog-triage-2026-06.md) § T13

> Raw ask: "saying we should never merge is a lie - we need a window for
> merging, done by the supervisor: insights and suggestions about merging
> plan (FF, merge, rebase...), force a feat branch when needed (configurable,
> no hard rules), a deeper Analyze for whether merging would hurt/break/risk
> functionality. Advising for a non-developer and for a developer."

## One-line summary

A read-only advisory layer over the existing integration machinery: a Merge
window (dashboard page + CLI) that shows, per merge-ready run, the branch
topology, the dry-run conflict report, the honest assurance lanes, and a
deterministic recommendation with a persona-attributed explanation at two
depths. The merge itself remains the existing explicit, gated,
human-confirmed action. Nothing in this feature writes to git.

## What already exists (and stays the source of truth)

| Capability | Where |
|---|---|
| Merge-ready run list | `listMergeReadyRuns` - `src/integration/integration-service.ts:78` |
| Cumulative dry-run merge preview (scratch worktree, conflict files) | `mergePreview` - `src/integration/integration-service.ts:111` |
| Integration branch creation (forks from current main, never touches main, credit trailers) | `integrate` - `src/integration/integration-service.ts:168` |
| Gated merge to main (typed confirm, lock, TOCTOU re-checks, broker `git.merge`, API token) | `finishIntegration` - `src/integration/integration-service.ts:326` |
| Honest verdict: lanes + `anyRealCheckPassed` (T2) | `src/safety/run-assurance.ts` |
| Project continuity ledger (T9) | `src/core/project-ledger.ts` |
| Workspace surface (T1): worktree + branch per run | `vibe path`, RunDetail Workspace panel |
| Supervisor personas (lens presets, not voice) | `src/orchestrator/personas.ts` |

The advisor **consumes** these; it does not duplicate or replace any of them.

### How a merge actually lands today (the model the advisor must tell the truth about)

The user never merges a run branch into main directly. The path is:

1. `integrate` forks an integration branch **from current main**
   (`startPoint: baseBranch`, `integration-service.ts:209`) and merges the
   selected run branches into it.
2. `finishIntegration` runs `git merge --no-edit <integrationBranch>` on
   main (`integration-service.ts:460`) - **no `--no-ff`**. Because the
   integration branch forked from main, this **fast-forwards** whenever
   main has not moved between apply and finish. If main did move, git
   creates a real merge commit, or the merge conflicts and is aborted.

So the commit shape is not a strategy the user picks; it is determined by
whether main moved in the apply-to-finish window. The advisor's job is to
predict and explain that, not to offer a choice the primitive does not
have. Run-branch topology (ahead/behind main) is still computed and shown,
but explicitly as "how far this change has drifted from main" - a conflict
and staleness signal - never as the merge artifact's topology.

## Decisions

### D1. The recommendation is deterministic code, not LLM

The recommendation (`finish-now` / `stage-on-integration-branch` /
`resolve-first`) is computed by a pure function from observable git facts
and the assurance lanes. Same inputs, same advice. Rationale: this surface
advises writes to main; a model hallucinating "safe to merge" is the exact
failure mode T2 just removed from verdicts. The LLM appears only in the
optional "Analyze deeper" pass (D5), which is read-only, clearly labeled,
and can never upgrade the deterministic recommendation - only add caution.

### D2. The advisor reads lanes + `anyRealCheckPassed`, never the bare verdict

Per T2's contract: `verified` with `anyRealCheckPassed: false` means
"nothing needed checking", not "checked and approved". The advisor renders
that distinction explicitly ("No check actually exercised this change") and
treats it as a caution flag, not a green light. Lane gaps (validation
missing, review not approved, verification not run) each map to a named
risk flag. The projection imports the real lane-status unions from
`RunAssurance` - no widening to `string`; a renamed status must fail the
typecheck, not silently stop matching.

### D3. Advisory text is deterministic, two depths, one structure

Persona is a judgment-posture preset, not a voice layer
([`orchestrator-personas.md`](./orchestrator-personas.md)); we keep that.
The advisory text is generated by pure code from the advice object:

- `headline` - plain language for a non-developer. "Safe to merge: small
  change, both checks passed, applies cleanly." / "Hold on: this change
  touches protected files and review never approved it."
- `detail` - developer depth: the numbers (ahead/behind, files, conflicts),
  the lane statuses, the predicted commit shape, and the recommendation
  rationale.

One text, progressive disclosure (matches the docs voice rule): the UI
shows the headline with an expandable detail; the CLI prints both. The
active supervisor persona is attributed on the card and may reorder which
risk flags lead (security persona lists protected-path/secret flags first),
but a persona can never add, remove, or soften a flag. Manual git commands
(e.g. conflict resolution) appear only in the developer detail, never in
the non-developer headline.

### D4. No new write paths in slice 1

The advisor adds **zero** git-mutating surface. Apply/finish stay exactly
the existing actions with their existing gates. Honest statement of what
those gates are by default: the operative gates are the typed
`merge-to-main` confirmation (CLI) and the `VIBESTRATE_API_TOKEN`
requirement on the HTTP finish route, plus the lock, TOCTOU re-checks,
HEAD-on-main requirement, no push, no auto-merge. The action broker is
consulted with `git.merge`, but **no policy is seeded by default, so its
decision is `allow` out of the box** (`src/server/routes/integration.ts:90`
documents this); projects may add a `git.merge` policy to deny or hold.
This doc must not oversell the broker as a default protection.

### D5. "Analyze deeper" is an optional, read-only consult pass

A consult-style LLM pass over the diff vs main plus hot-path context:
files in this diff that other recent runs also touched (from run states +
ledger `shipped`/`residual` entries), and whether touched files fall inside
the configured `commands.validate` scope. Honest scope note: the triage
asked for "test coverage of touched files"; validate-scope membership is a
**proxy** for that, not real coverage data - a true per-file coverage map
is a separate capability, logged as a follow-up. Existing redaction rules
apply (no `.env` contents, secret-shaped lines refused). Output is markdown
rendered in the window and cached under the run's artifacts dir. It is
advisory prose; the deterministic recommendation and flags are computed
before and independent of it. Exposure note: the endpoint triggers a local
provider invocation, the same exposure class as the existing consult
endpoints on the same bind - no new class of exposure, but stated here
honestly.

### D6. Branch policy is suggestion-only, configurable

The raw ask says "configurable, no hard rules". A `merge.advisor` config
section sets thresholds above which the advisor recommends staging on an
integration branch (apply now, hold finish) instead of finishing promptly.
Crossing a threshold changes the *suggestion*, never blocks the action.
Existing hard policies (`forbidAutoMerge`, `forbidAutoPush`,
`requireHumanMerge`) are untouched.

### D7. Rebase advice is deferred entirely

An earlier draft recommended `rebase-then-merge` when project history
looked linear (merge-commit count heuristic). Dropped from this design:
the heuristic is unsound (one historic merge commit flips it; squash-merge
conventions read as "linear" and would get rebase advice the team does not
want), the existing finish path already fast-forwards in the common case
so the linear-history motivation mostly evaporates, and rebase manual
steps rewrite the run branch - the riskiest instruction the advisor could
hand a non-developer. If demand appears, it returns behind an explicit
config opt-in, never a heuristic.

## Data model

New module `src/integration/merge-advisor.ts`:

```ts
type MergeRecommendation =
  | "finish-now"                  // apply + finish via the existing actions
  | "stage-on-integration-branch" // apply, but hold finish: large, protected, or multi-run
  | "resolve-first";              // preview conflicted: resolve before merging

type MergeRiskFlag = {
  id:
    | "preview_conflict"
    | "validation_gap" | "review_gap" | "verification_gap"
    | "no_real_check"             // verdict verified but anyRealCheckPassed=false
    | "protected_paths"
    | "large_change"              // filesTouched > threshold
    | "diverged_main"             // run branch far behind main: stale change
    | "overlaps_other_ready";     // cumulative preview shows ordering effects
  severity: "caution" | "warning";
  summary: string;                // one plain-language line
  detail: string;                 // developer depth
};

type BranchTopology = {
  branchName: string;             // the RUN branch - drift signal, not the merge artifact
  aheadOfMain: number;            // rev-list --left-right --count
  behindMain: number;
  filesTouched: number;           // diff --name-only vs merge-base
  protectedPathHits: string[];
};

type MergeAdvice = {
  runId: string;
  topology: BranchTopology;
  preview: BranchPreview | null;  // existing type; null in the cheap list projection (see Surfaces)
  assurance: {                    // projection using the real lane-status unions from RunAssurance
    verdict: RunAssuranceVerdict;
    lanes: {
      validation: ValidationLaneStatus;
      review: ReviewLaneStatus;
      verification: VerificationLaneStatus;
    };
    anyRealCheckPassed: boolean;
  };
  recommendation: MergeRecommendation;
  recommendationReason: string;
  predictedShape: "fast-forward" | "merge-commit-if-main-moves";
  flags: MergeRiskFlag[];
  headline: string;               // plain-language advisory (D3)
  detail: string;                 // developer advisory (D3)
  personaId: string;              // attribution only
  manualSteps: string[] | null;   // git commands (developer detail only), e.g. conflict resolution
};
```

`computeMergeAdvice(input): MergeAdvice` is pure: it takes the topology
facts, the preview result, the assurance projection, the ledger state, the
persona id, and the `merge.advisor` config, and returns the advice. A thin
collector (`collectBranchTopology`) gathers git facts read-only
(`rev-list`, `diff --name-only`); `mergePreview` is reused as-is for
conflict facts (it already cleans up its scratch worktree).

### Recommendation rules (initial, all deterministic)

1. Preview conflicted -> `resolve-first`. Developer detail names the
   conflicted files and suggests resolving on an integration branch
   (`integrate apply` stops at the conflict and leaves the worktree
   mergeable by design).
2. Any threshold from D6 crossed (files touched, protected paths, multi-run
   overlap) -> `stage-on-integration-branch`: apply now, validate on the
   integration branch, finish deliberately later.
3. Otherwise -> `finish-now` via the existing apply + finish path.

`predictedShape` is rendered alongside: "this will fast-forward main" when
nothing else is in flight, with the caveat that a merge commit (or a
refusal on conflict) results if main moves between apply and finish - the
same window the existing TOCTOU `tipSha` re-check guards.

Assurance gaps never change the recommendation - they are flags. The
recommendation answers "how should this land"; flags answer "should it
land at all". The window renders flags above the recommendation so the
caution reads first.

## Surfaces

### Dashboard: Merge page (new route `kind: "merge"`)

Hub-style, following the flows-hub pattern. Two levels with different
costs (`mergePreview` does real scratch-worktree git work per call; the
hub must not fan that out on page load):

- **List**: merge-ready runs (reusing `GET /api/integration`) with the
  *cheap* projection only - assurance lanes + run-branch topology
  (`rev-list`/`diff` counts are fast read-only ops). `preview: null` here.
- **Merge window** (drill-in, per run): full advice. Risk flags (plain
  headline, expandable developer detail) with persona attribution;
  assurance lanes; topology card; dry-run conflict report; recommendation
  + reason + predicted commit shape; the existing preview / apply / finish
  actions with their existing confirmation UX (finish keeps the typed
  confirmation + token requirement). "Analyze deeper" button (slice 2).

The merge-ready section currently inlined in RunsPage links here; it can
slim down once the page ships (follow-up, not a slice-1 requirement).

### CLI: `vibe integrate advise [runIds...] [--json]`

Prints the full advice per named run: flags, lanes, topology, predicted
shape, recommendation, manual steps. With no runIds, advises all
merge-ready runs (accepting the preview cost explicitly, same as
`vibe integrate preview`). `--json` emits the `MergeAdvice[]` verbatim.
UI and CLI share `computeMergeAdvice` (parity by construction).

### API: `POST /api/integration/advice`

Body `{ runIds?: string[] }` (mirrors `POST /api/integration/preview`,
which it wraps). Returns `{ advice: MergeAdvice[] }`. POST because it does
scratch-worktree git work like preview; it mutates no branches. Gating
matches preview exactly: open on a tokenless loopback bind, bearer-gated
when a token is configured - advice is the same read-only class and the
same cost class as preview, and this doc states that exposure rather than
hiding it. No other new endpoints.

## Config (slice 3)

```yaml
merge:
  advisor:
    suggestIntegrationBranchWhen:
      filesTouched: 25        # suggest staging above this many changed files
      protectedPaths: true    # any policies.protectedPaths hit
      behindMain: 50          # run branch heavily diverged from main
```

Defaults conservative; all three only flip the recommendation to
`stage-on-integration-branch`. Schema added to
`src/project/config-schema.ts` with `.default({...})` for existing
projects; editable via `vibe config set` like any section.

## Security invariants (unchanged, restated for review)

- No auto-merge, no push, no scheduler path to `git.merge` - all preserved;
  this feature adds no mutating endpoint or command.
- `finishIntegration` gates untouched: typed `merge-to-main` confirmation,
  HTTP token requirement, lock, TOCTOU re-checks, HEAD-on-main requirement.
  The broker `git.merge` decision is recorded but is `allow` by default
  (no seeded policy) - the typed confirmation and token are the operative
  default gates (see D4).
- Advisor git operations are read-only (`rev-list`, `diff --name-only`)
  plus the existing self-cleaning `mergePreview`.
- LLM output (slice 2) can only add prose caution; it cannot change
  `recommendation`, `flags`, or any action gating. It is clearly labeled
  as model-generated in the UI.
- Diff content shown or sent to the model follows existing redaction rules:
  secret-shaped added lines and secret-like paths are refused or redacted;
  `.env` contents never read.
- All file reads stay path-guarded to project root + run worktrees.

## Slices

1. **(1a) Advisor core + CLI (S-M):** `merge-advisor.ts` (pure compute +
   topology collector), `POST /api/integration/advice`,
   `vibe integrate advise`. Tests: pure-function table tests for rules and
   flag derivation (including the `no_real_check` projection);
   temp-git-repo smoke for the topology collector and for predicted-shape
   truthfulness (integration branch ff's when main is unmoved, merge
   commit when main moves); route-level test for the endpoint.
2. **(1b) Merge page (S-M):** route `kind: "merge"`, hub list (cheap
   projection) + merge window (full advice), existing actions embedded.
   Typecheck/build + browser click-through.
3. **(2) Analyze deeper (S-M):** consult-style read-only pass (D5), cached
   artifact, fake-provider tests.
4. **(3) Branch policy config (S):** `merge.advisor` schema + thresholds
   wired into rule 2, config docs, `vibe config set` round-trip test.

Each slice ships with docs (`docs/content/` merge workflow page update,
`pnpm docs:generate`) and a changelog entry per repo convention.

## Open questions

- Should `finishIntegration` later grow a `--ff-only` mode (refuse instead
  of creating a merge commit when main moved)? Deferred: it widens the most
  sensitive write path; today's behavior is already ff-first.
- Should advice be persisted into the ledger (`decision` entry when the
  user merges against a warning)? Deferred to T18's measurement work.
- Real per-file test-coverage data for "Analyze deeper" (replacing the
  validate-scope proxy). Separate capability; not scheduled.
- TUI shell surface: the Ink RunsPage could show the headline per
  merge-ready run. Nice-to-have after slice 1.
- Rebase advice behind explicit config opt-in (see D7), only if demand
  appears.
